# Материалы для пояснительной записки

## 1. Предлагаемое название

«Разработка веб-приложения для создания и чтения интерактивных комиксов с ветвящимся сюжетом и AI-генерацией иллюстраций».

Альтернативный вариант:

«Проектирование и реализация информационной системы InkFlow Comics для публикации интерактивных графических историй».

## 2. Актуальность

Обычный цифровой комикс сохраняет линейную структуру печатного издания и почти не использует интерактивные возможности веб-платформы. Пользователь может перелистывать страницы, но не влияет на развитие сюжета. Авторам при этом приходится отдельно использовать графические редакторы, файловые хранилища, системы публикации и инструменты для проектирования нелинейного сценария.

Разработанное приложение объединяет эти процессы в одной системе. Автор создаёт страницы, связывает их вариантами выбора, загружает или генерирует иллюстрации, проверяет историю в режиме предварительного просмотра и публикует её. Читатель проходит историю по выбранному маршруту, сохраняет прогресс, добавляет произведения в закладки и оставляет рецензии.

## 3. Цель

Разработать полнофункциональное адаптивное веб-приложение, позволяющее создавать, публиковать и читать интерактивные комиксы с ветвящимся сюжетом, анимированными переходами и AI-генерацией изображений.

## 4. Объект и предмет

Объект исследования: цифровые платформы создания и распространения интерактивного визуального контента.

Предмет исследования: методы представления ветвящегося сюжета в виде ориентированного графа, организация клиент-серверного взаимодействия и интеграция генеративной модели изображений в редактор комиксов.

## 5. Задачи

1. Проанализировать предметную область и определить роли автора и читателя.
2. Сформировать функциональные и нефункциональные требования.
3. Спроектировать клиент-серверную архитектуру и реляционную базу данных.
4. Реализовать регистрацию, вход и обновление JWT-токенов.
5. Реализовать редактор страниц и переходов ветвящегося сюжета.
6. Реализовать drag-and-drop визуализацию графа.
7. Реализовать загрузку и AI-генерацию иллюстраций.
8. Реализовать каталог, фильтрацию, чтение, закладки, прогресс и рецензии.
9. Обеспечить адаптивность для desktop, планшетов и смартфонов.
10. Добавить 20 вариантов анимированных переходов.
11. Подготовить демонстрационные данные из 12 комиксов с четырьмя финалами.
12. Выполнить unit, component и e2e тестирование.
13. Выполнить аудит безопасности и оптимизацию production bundle.

## 6. Практическая значимость

Система может использоваться авторами, образовательными организациями и небольшими издательствами для создания интерактивных историй без разработки отдельного сайта под каждое произведение. Архитектура допускает дальнейшее развитие до SaaS-платформы, корпоративного конструктора обучающих сценариев или инструмента визуальных новелл.

## 7. Пользователи и сценарии

Автор:

- регистрируется и входит в систему;
- создаёт черновик;
- задаёт название, описание, сценарий и обложку;
- создаёт страницы и панели;
- задаёт стартовую страницу;
- связывает страницы вариантами выбора;
- проверяет граф и предварительный просмотр;
- загружает изображения или создаёт 1–4 AI-варианта;
- публикует комикс.

Читатель:

- просматривает каталог;
- использует фильтры и сортировку;
- открывает описание произведения;
- читает историю, выбирая направление сюжета;
- возвращается к предыдущему выбору;
- сохраняет прогресс;
- добавляет комикс в закладки;
- выставляет оценку и пишет рецензию.

## 10. Архитектура

Использована трёхуровневая клиент-серверная архитектура:

1. Presentation layer: React SPA.
2. Application layer: Fastify REST API.
3. Data layer: PostgreSQL и Prisma ORM.

Клиент не обращается к базе данных напрямую. Все операции проходят через REST API. Авторизация защищённых endpoint выполняется JWT access token. Для продолжения сессии используется refresh token.

Основные каталоги:

- `client/src/pages` — экраны;
- `client/src/features` — предметные API и контексты;
- `client/src/components` — переиспользуемые компоненты;
- `server/src/modules` — маршруты предметных модулей;
- `server/src/plugins` — Fastify plugins;
- `server/prisma` — схема, миграции и seed;
- `server/uploads` — локальные изображения.

## 11. Выбор технологий

React выбран для компонентной разработки SPA и эффективного обновления интерфейса.

TypeScript обеспечивает статический контроль контрактов между клиентом, API и моделью данных.

Mantine предоставляет доступные формы, сетку, модальные элементы и адаптивные компоненты.

Framer Motion используется для переходов страниц и drag-and-drop узлов графа.

TanStack Query управляет серверным состоянием, кешированием и инвалидированием запросов.

Fastify выбран за низкие накладные расходы, plugin-архитектуру и встроенный механизм inject для тестирования API.

Prisma обеспечивает типобезопасную работу с PostgreSQL и воспроизводимые миграции.

Zod используется для проверки тела запроса, query-параметров и переменных окружения.

Vitest, Testing Library и Playwright покрывают модульный, компонентный и браузерный уровни.

## 12. Модель данных

`User`:

- учётные данные;
- профиль;
- связи с комиксами, закладками, прогрессом и рецензиями.

`Comic`:

- метаданные произведения;
- статус `DRAFT` или `PUBLISHED`;
- автор;
- страницы.

`ComicPage`:

- уникальный внутри комикса `pageKey`;
- позиция;
- текст;
- основное изображение;
- изображения панелей;
- prompt;
- стиль перехода;
- признак стартовой страницы.

`ComicChoice`:

- исходная страница;
- текст выбора;
- целевая страница.

`ComicChoice` превращает набор страниц в ориентированный граф. Одна страница может иметь несколько исходящих и входящих рёбер.

`ComicBookmark`, `ReadProgress`, `ComicReview` реализуют пользовательские функции чтения.

## 13. Алгоритм сохранения графа

1. Zod проверяет поля и ограничения.
2. Система нормализует стартовую страницу.
3. Проверяется уникальность `pageKey`.
4. Для каждого выбора проверяется существование `targetPageKey`.
5. В транзакции удаляется старый граф.
6. Создаются новые страницы.
7. Формируется отображение `pageKey -> UUID`.
8. Создаются рёбра `ComicChoice`.

Такой подход исключает переходы на несуществующую страницу и частично сохранённый граф.

## 14. Алгоритм чтения

Клиент получает массив страниц и `startPageId`, создаёт Map по UUID и показывает текущую страницу. При выборе:

1. текущий UUID помещается в историю;
2. определяется направление анимации;
3. применяется стиль исходной страницы;
4. открывается целевая страница;
5. для авторизованного пользователя сохраняется прогресс.

Кнопка «Назад» извлекает последний UUID из локальной истории маршрута.

## 15. AI-генерация

Маршрут `POST /api/uploads/generate`:

1. требует JWT;
2. ограничен шестью запросами в минуту на пользователя;
3. проверяет prompt, размер и число вариантов;
4. отправляет запрос в OpenAI Images API;
5. принимает base64 или временный URL;
6. проверяет сигнатуру ответа;
7. сохраняет файл под случайным UUID;
8. возвращает локальный URL.

Поддерживается 1–4 варианта изображения. В prompt автоматически добавляются выбранный стиль и постоянные признаки персонажей. При safety-блокировке сервер выполняет один повтор с нейтральным family-friendly описанием без реальных имён, логотипов и существующих персонажей.

Диагностические коды:

- `OPENAI_NOT_CONFIGURED`;
- `OPENAI_INVALID_KEY`;
- `OPENAI_ACCESS_DENIED`;
- `OPENAI_QUOTA`;
- `OPENAI_TIMEOUT`;
- `OPENAI_EMPTY_RESPONSE`.

Практическая проверка выполнена реальным запросом: HTTP 200, PNG сохранён в `server/uploads/comics`.

## 17. Демонстрационные данные

- 17 страниц в каждом;

Десять классических историй используют отдельные public-domain выпуски Planet Comics и Blue Beetle из Internet Archive. Каждый комикс получил 17 собственных сканов без пересечений с другими историями. SVG-заглушки больше не используются в редакторе и demo-историях. Полный перечень: `ASSET_SOURCES.md`.

В набор включены две трансформативные фанатские истории: завершённый Batman-комикс и комикс по мотивам сериала `The Boys`. Для них написаны отдельные тексты всех 17 сцен. Официальные страницы DC и кадры Amazon не копируются: Batman использует CC-лицензированные cosplay-фотографии, а The Boys — CC-лицензированные фотографии актёрского состава и доступный по CC материал со Старлайт в образе.

## 19. Безопасность

Реализовано:

- bcrypt для паролей;
- короткоживущий access token;
- отдельный секрет refresh token;
- Helmet и CSP;
- строгий CORS;
- Zod-валидация;
- ограничения body и multipart;
- случайные имена файлов;
- проверка magic bytes JPEG, PNG, GIF и WEBP;
- запрет произвольных схем image URL;
- rate limiting регистрации, входа, refresh, upload и AI;
- отсутствие подробных stack trace в ответе;
- request ID;
- таймаут внешнего AI-запроса;
- dependency audit.

Исходный аудит обнаружил 15 уязвимостей: 3 critical, 3 high, 8 moderate и 1 low. После обновления `@fastify/jwt`, `@fastify/static`, Vite, React Router, Axios и точечного override `esbuild` команда `pnpm audit` сообщает `No known vulnerabilities found`.

Остаточный риск MVP: refresh token хранится в web storage. Для production рекомендуется HttpOnly Secure SameSite cookie, ротация refresh token и таблица серверных сессий.

## 26. Готовые выводы

В ходе работы была спроектирована и реализована клиент-серверная система для создания интерактивных комиксов. В отличие от линейного просмотра, разработанная модель представляет произведение ориентированным графом, в котором пользовательский выбор определяет следующую страницу.

Приложение реализует полный цикл работы с контентом: создание, визуальное проектирование ветвей, загрузку и генерацию изображений, публикацию, чтение, сохранение прогресса и обратную связь.

Проведённое тестирование подтвердило корректность ключевых алгоритмов и работоспособность интерфейса на desktop и mobile. Аудит зависимостей и дополнительные серверные ограничения повысили безопасность MVP. Применение ленивой загрузки и code splitting существенно сократило стартовый JavaScript bundle.

Поставленная цель достигнута, а архитектура допускает дальнейшее масштабирование до многопользовательской платформы с объектным хранилищем, совместным редактированием и промышленной системой наблюдаемости.
